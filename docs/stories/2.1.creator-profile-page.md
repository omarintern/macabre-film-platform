# Story 2.1: Creator Profile Page (Firebase Version)

## Status
Done

## Story
**As a** Creator or Admin, **I want** a dedicated public profile page, **so that** I have a professional space to showcase my works and information.

## Acceptance Criteria
1. Authenticated users with 'CREATOR' or 'ADMIN' role can access a page to edit their profile information.
2. A user's profile is publicly viewable at a unique URL for other authenticated users.
3. The public profile displays the user's name, bio, and a section for their submitted works.
4. Profile editing is restricted to the profile owner only.
5. All profile operations use Firebase Firestore directly (no API routes).

## Dev Notes

### Firebase Architecture
**100% Client-Side Implementation**:
- No API routes needed - all data operations use Firebase Firestore directly
- Authentication handled by Firebase Auth client-side
- Profile data stored in Firestore `users` collection
- Real-time updates possible with Firebase listeners

### Data Models
**Firebase User Document Structure**:
```typescript
interface User {
  id: string;
  email: string;
  role: 'CREATOR' | 'ADMIN';
  name?: string;
  bio?: string;
  createdAt: Timestamp;
  updatedAt: Timestamp;
}
```

### Component Specifications
**Client-Side Components**:
- All profile pages use `'use client'` directive
- Use `useAuth()` hook from userSessionStore for authentication
- Use `firebaseDataService` for all data operations
- Implement proper loading states and error handling

### File Locations
- Profile edit page: `app/(main)/profile/edit/page.tsx` (client component)
- Public profile page: `app/(main)/profile/[userId]/page.tsx` (client component)
- Profile redirect page: `app/(main)/profile/page.tsx` (client component)
- Profile components: `components/features/ProfileEditForm.tsx`, `components/features/PublicProfile.tsx`
- Firebase data service: `lib/firebase/dataService.ts`

### Authentication Flow
1. User navigates to `/profile` → redirects to `/profile/${userId}`
2. User navigates to `/profile/edit` → shows edit form if authenticated
3. User navigates to `/profile/[userId]` → shows public profile if authenticated
4. All authentication handled client-side using Firebase Auth

### Security Considerations
- Profile editing restricted to profile owner only
- Public profiles only visible to authenticated users
- Input validation and sanitization for name and bio fields
- Proper error handling to prevent information leakage

### Testing Requirements
- Component tests for profile edit form
- Component tests for public profile display
- Integration tests for profile workflow end-to-end
- Firebase authentication state testing
